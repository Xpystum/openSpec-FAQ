Spec-Driven Development для AI-ассистентов
OpenSpec — лёгкий фреймворк, который заставляет тебя и AI договориться о том, что строить, прежде чем будет написана хоть одна строка кода. Спецификации живут в репозитории вместе с кодом.

Философия

Fluid, не rigid
Без жёстких фаз — обновляй что угодно, когда угодно
Iterative, не waterfall
Маленькие шаги, быстрая обратная связь
Brownfield-first
Сделан для живых проектов, не только для нового кода
Easy, не complex
Минимум процесса, максимум пользы

Три проблемы которые решает

Проблема 1 Контекст умирает вместе с чатом
AI не помнит что обсуждалось вчера. OpenSpec фиксирует договорённости в файлах, которые живут в репозитории постоянно.
Проблема 2 AI пишет не то что нужно
Без спецификации AI интерпретирует задачу по-своему. Spec — точка выравнивания между человеком и AI перед началом работы.
Проблема 3 Ревью кода без понимания зачем
Reviewer смотрит на код, но не знает что менялось в требованиях. Spec delta показывает это явно — до просмотра кода.

Установка и первый запуск

Требуется Node.js 20.19.0 или выше. Также работает через pnpm, yarn, bun и nix.

1. Установить глобально

npm install -g @fission-ai/openspec@latest

2. Инициализировать в проекте

cd your-project openspec init

В процессе init тебя спросят какой AI-инструмент использовать (Claude Code, Cursor, Windsurf и т.д.). OpenSpec создаст нужные конфиги автоматически.

3. Запустить онбординг (опционально)

/opsx:onboard

Интерактивный тьюториал на реальной кодовой базе. 11 фаз, ~15 минут. Рекомендован для первого знакомства.

Обновление

# Обновить пакет npm install -g @fission-ai/openspec@latest # Регенерировать инструкции AI в проекте openspec update
После обновления пакета всегда запускай openspec update внутри каждого проекта — это обновляет skills и slash-команды для AI-инструмента.

Файлы и папки

После openspec init появляются две зоны: папка openspec/ для спецификаций и папка AI-инструмента для skills и команд.

Полная структура

your-project/ ├── openspec/ │ ├── config.yaml ← стек, соглашения, правила проекта │ ├── specs/ ← живые спецификации (source of truth) │ │ ├── auth/ │ │ │ └── spec.md ← спека домена "auth" │ │ └── payments/ │ │ └── spec.md │ └── changes/ ← активные изменения │ ├── add-dark-mode/ ← папка одного изменения │ │ ├── .openspec.yaml ← метаданные (схема, статус) │ │ ├── proposal.md ← зачем и что │ │ ├── design.md ← техническое решение │ │ ├── tasks.md ← чеклист задач [x]/[ ] │ │ └── specs/ ← дельта требований │ │ └── ui/spec.md │ └── archive/ ← завершённые изменения │ └── 2026-03-31-add-dark-mode/ │ └── .claude/ ← (если выбран Claude Code) ├── skills/ ← авто-детектируемые навыки │ ├── openspec-propose/SKILL.md │ ├── openspec-explore/SKILL.md │ ├── openspec-apply-change/SKILL.md │ └── openspec-archive-change/SKILL.md └── commands/opsx/ ← slash-команды ├── propose.md ├── explore.md ├── apply.md └── archive.md

Артефакты — подробно

proposal.md — Зачем и что делаем

Создаётся первым. Определяет намерение и границы задачи. Ключевой артефакт для выравнивания понимания.

# Proposal: Add Dark Mode ## Intent Пользователи просят тёмную тему для снижения нагрузки на глаза. ## Scope ### In scope - Переключатель темы в настройках - Определение системных предпочтений (prefers-color-scheme) - Сохранение выбора в localStorage ### Out of scope - Тёмная тема для email-уведомлений (отдельная задача) - Мобильное приложение ## Approach CSS custom properties + React Context для управления состоянием. ## Open questions - Нужна ли анимация при переключении темы?

specs/*.md — Дельта требований

Хранит только изменения относительно текущих specs. Маркеры: ADDED / MODIFIED / REMOVED. При archive — мержится в основные specs.

Формат The system SHALL... — стандарт IEEE 830. SHALL = обязательно, SHOULD = рекомендуется, MAY = опционально.
# Delta for UI ## ADDED Requirements ### Requirement: Theme Selection The system SHALL allow users to choose between light and dark themes. #### Scenario: User switches to dark mode - GIVEN пользователь открыл настройки - WHEN выбирает "Dark mode" - THEN интерфейс переключается на тёмную тему - AND выбор сохраняется между сессиями #### Scenario: System preference detection - GIVEN prefers-color-scheme: dark в ОС - WHEN пользователь первый раз открывает приложение - THEN применяется тёмная тема автоматически

design.md — Технические решения

Архитектурный подход, выбор инструментов, компромиссы. Необязателен для простых задач.

# Design: Add Dark Mode ## Approach CSS custom properties для тем, React Context для состояния. ## Key Decisions - **CSS variables, не CSS-in-JS** — нет runtime overhead, нативная поддержка system preference через prefers-color-scheme - **localStorage, не backend** — не требует auth, мгновенный отклик ## Sequence User clicks toggle → ThemeContext.toggle() → localStorage.set('theme', value) → CSS class on <html> → Все компоненты реагируют через CSS vars

tasks.md — Чеклист реализации

AI читает этот файл при /opsx:apply и выполняет задачи по порядку, ставя галочки.

# Tasks: Add Dark Mode ## Phase 1: Core Infrastructure - [ ] 1.1 Создать ThemeContext с toggle() и defaultTheme - [ ] 1.2 Добавить CSS custom properties в globals.css - [ ] 1.3 Обернуть App в ThemeProvider ## Phase 2: UI Components - [ ] 2.1 Создать компонент ThemeToggle (иконка sun/moon) - [ ] 2.2 Добавить ThemeToggle в Header - [ ] 2.3 Подключить localStorage persistence ## Phase 3: Tests - [ ] 3.1 Unit-тест для ThemeContext - [ ] 3.2 E2E: переключение темы сохраняется после reload

Команды OPSX

Core-профиль (активен по умолчанию)

КомандаРольЧто делаетКогда
/opsx:explore Thinking Режим обсуждения без генерации артефактов. AI изучает кодовую базу, задаёт уточняющие вопросы. Код не пишется. Перед началом, для проработки ТЗ
/opsx:propose <name> Планирование Создаёт папку изменения + сразу все 4 артефакта: proposal, specs, design, tasks за один шаг. Когда понимание есть, нужно создать всё быстро
/opsx:apply [name] Реализация Читает tasks.md, выполняет задачи по порядку, ставит [x]. Если name не указан — определяет из контекста. После ревью артефактов
/opsx:archive [name] Завершение Мержит delta-specs в openspec/specs/. Перемещает папку в archive/ с датой. После успешного verify

Expanded-профиль (дополнительно)

КомандаРольЧто делает
/opsx:new <name>ЗаготовкаТолько создаёт папку и .openspec.yaml. Артефакты не генерирует — первый шаг пошагового режима.
/opsx:continueШагСоздаёт один следующий незаблокированный артефакт. Читает граф зависимостей и берёт следующий доступный.
/opsx:ff [name]Fast-forwardГенерирует все оставшиеся артефакты сразу. Используется после new.
/opsx:verifyВалидацияПроверяет: все задачи выполнены, код соответствует specs, design-решения отражены в коде.
/opsx:syncСинхронизацияОбновляет specs если реализация пошла не по плану. Актуализирует дельту.
/opsx:bulk-archiveПакетАрхивирует несколько завершённых изменений сразу. Автоматически разрешает конфликты в specs по хронологии.
/opsx:onboardОбучениеИнтерактивный тьюториал на реальной кодовой базе. 11 фаз, ~15 минут.

Сценарий: пришло ТЗ → спецификация → код

1
Инициализация (один раз на проект)
openspec init → выбираешь AI-инструмент → создаётся вся структура и skills
2
/opsx:explore — вставляешь ТЗ
AI читает кодовую базу, задаёт уточняющие вопросы, выясняет edge cases. Никакого кода ещё нет. Только выравнивание понимания.
3
/opsx:propose user-auth — создаём все артефакты
AI использует контекст из explore-сессии. Создаёт proposal, specs, design, tasks за один шаг.
4
Ревью артефактов руками
Открываешь proposal.md, specs, tasks — читаешь, правишь. Просишь AI уточнить конкретный артефакт. Это самый важный этап.
5
/opsx:apply — реализация
AI идёт по tasks.md и выполняет пункт за пунктом. Ставит [x]. Перед этим шагом — очисти контекст чата.
6
/opsx:verify → /opsx:archive
Verify проверяет соответствие кода specs. Archive мержит дельту в основные specs и кладёт изменение в архив.
Делай explore и propose в одной сессии — AI использует живой контекст разговора. Если начинаешь новую сессию — кратко пересказывай что решили в explore.

Параллельная работа над несколькими задачами

# Задача 1 прервана → начали срочный фикс /opsx:new fix-urgent-bug /opsx:ff /opsx:apply /opsx:archive # Возвращаемся к задаче 1 /opsx:apply feature-dark-mode ← AI подхватит с того места где остановился # Завершили несколько задач → архивируем пакетом /opsx:bulk-archive

openspec/config.yaml

Главный файл конфигурации проекта. Инжектируется в каждый запрос к AI при генерации артефактов. Именно здесь OpenSpec узнаёт о стеке, архитектуре и соглашениях.

Три слоя

КлючЧто делаетКуда инжектируется
schemaДефолтная схема workflow. Убирает --schema из каждой команды.При каждой команде
contextСтек, архитектура, соглашения. «Мировоззрение» проекта. Макс. 50KB.Во ВСЕ артефакты
rulesПравила для конкретных артефактов — proposal, specs, design, tasks.Только в нужный артефакт

Полный пример (Laravel-проект)

schema: spec-driven context: | Language: Russian. All artifacts must be written in Russian. Tech stack: PHP 8.3, Laravel 11, Vue 3 + Inertia.js Database: PostgreSQL, Eloquent ORM API style: REST JSON API + web routes через Inertia Auth: Laravel Sanctum, роли через Spatie Permission Testing: Pest для unit/feature, Playwright для e2e Style: Laravel Pint, strict PHP types Architecture: Service Layer поверх MVC Conventions: - Backwards compatibility обязательна для всех public API - Все новые эндпоинты — FormRequest обязателен - Feature-тест обязателен для каждого нового эндпоинта rules: proposal: - Указывать нужны ли миграции и сидеры - Указывать план отката для рискованных изменений - Отмечать если изменение затрагивает API-контракт specs: - Формат Given/When/Then для всех сценариев - Покрывать авторизацию (Policy/Gate) для каждого действия - Указывать HTTP-метод и маршрут для API-эндпоинтов design: - Указывать какие модели/relations затрагиваются - Описывать структуру миграций при изменениях схемы БД tasks: - Порядок: миграция → модель → сервис → контроллер → тест - Максимум 2 часа на одну задачу

Часто задаваемые вопросы

Это же просто waterfall?▶
Нет. Waterfall ломается из-за жёстких фаз и месяцев планирования. OpenSpec — это 10 минут на alignment перед началом. Когда что-то меняется — обновляешь spec и продолжаешь. Никаких фазовых ворот, никаких «нельзя вернуться назад».
Чем отличается от встроенного plan mode в AI?▶
Plan mode — для одной сессии. OpenSpec — для планов которые живут несколько сессий, шерятся с командой и проходят через весь lifecycle разработки. Контекст не умирает когда закрываешь вкладку.
Можно ли на существующем (brownfield) проекте?▶
Да, именно для этого и создан. Specs создаются по мере появления новых задач. Не нужно описывать весь существующий код сразу — это бессмысленная трата времени. Начни с первой новой задачи.
Что если поменял AI-инструмент?▶
Specs и changes в openspec/ работают с любым инструментом — они просто Markdown-файлы. Запусти openspec update для нового инструмента и получишь нужные skills и slash-команды.
Нужен ли API-ключ или MCP-сервер?▶
Нет. OpenSpec работает через slash-команды и файлы в репозитории. Никаких API-ключей, никаких MCP-серверов — просто npm-пакет и Markdown-файлы.
Как командная работа?▶
Specs и changes живут в репозитории — коллаборация через git: PR, reviews, стандартный workflow. Reviewer видит дельту требований в PR ещё до того как смотрит на код.
Какую модель использовать?▶
Официальная рекомендация: Claude Opus 4.5 или GPT 5 для планирования и реализации. Качество артефактов напрямую зависит от мощи модели. Слабые модели дают размытые specs.
OpenSpec собирает данные о проекте?▶
Только имена команд и версию пакета — для понимания паттернов использования. Никаких аргументов, путей, содержимого файлов и PII. Отключить: export OPENSPEC_TELEMETRY=0
Я vibe-кодер — это для меня?▶
Честный ответ: зависит. Если ищешь магический инструмент который всё сделает сам — не то. Specs работают только если ты их читаешь, думаешь и работаешь с ними. Встречай инструмент на полпути.
