Spec-Driven Development для AI-ассистентов
OpenSpec — лёгкий фреймворк, который заставляет тебя и AI договориться о том, что строить, прежде чем будет написана хоть одна строка кода. Спецификации живут в репозитории вместе с кодом.

Философия

Fluid, не rigid
Без жёстких фаз — обновляй что угодно, когда угодно
Iterative, не waterfall
Маленькие шаги, быстрая обратная связь
Brownfield-first
Сделан для живых проектов, не только для нового кода
Easy, не complex
Минимум процесса, максимум пользы

Три проблемы которые решает

Проблема 1 Контекст умирает вместе с чатом
AI не помнит что обсуждалось вчера. OpenSpec фиксирует договорённости в файлах, которые живут в репозитории постоянно.
Проблема 2 AI пишет не то что нужно
Без спецификации AI интерпретирует задачу по-своему. Spec — точка выравнивания между человеком и AI перед началом работы.
Проблема 3 Ревью кода без понимания зачем
Reviewer смотрит на код, но не знает что менялось в требованиях. Spec delta показывает это явно — до просмотра кода.

Установка и первый запуск

Требуется Node.js 20.19.0 или выше. Также работает через pnpm, yarn, bun и nix.

1. Установить глобально

npm install -g @fission-ai/openspec@latest

2. Инициализировать в проекте

cd your-project openspec init

В процессе init тебя спросят какой AI-инструмент использовать. OpenSpec создаст нужные конфиги автоматически.

3. Запустить онбординг (рекомендуется)

/opsx:onboard

Интерактивный тьюториал на реальной кодовой базе. 11 фаз, ~15 минут.

Обновление

# Обновить пакет npm install -g @fission-ai/openspec@latest # Регенерировать инструкции AI в проекте openspec update
После обновления пакета всегда запускай openspec update внутри каждого проекта — обновляет skills и slash-команды.

Рекомендуемые модели

ЗадачаРекомендацияПочему
Планирование (explore, propose)Claude Opus 4.5 / GPT 5Качество specs зависит от мощи модели
Реализация (apply)Claude Opus 4.5 / GPT 5Сложные задачи требуют глубокого контекста
Простые правкиЛюбая модель—
Перед /opsx:apply очищай контекст чата — OpenSpec работает лучше с чистым окном.

Файлы и папки

После openspec init появляются две зоны: папка openspec/ для спецификаций и папка AI-инструмента для skills и команд.

Полная структура

your-project/ ├── openspec/ │ ├── config.yaml ← стек, соглашения, правила проекта │ ├── specs/ ← живые спецификации (source of truth) │ │ ├── auth/spec.md ← спека домена "auth" │ │ └── payments/spec.md │ └── changes/ ← активные изменения │ ├── add-dark-mode/ │ │ ├── .openspec.yaml ← метаданные (схема, статус) │ │ ├── proposal.md ← зачем и что │ │ ├── design.md ← техническое решение │ │ ├── tasks.md ← чеклист [x]/[ ] │ │ └── specs/ui/spec.md ← дельта требований │ └── archive/ ← завершённые изменения с датой │ └── .claude/ ← конфиги для Claude Code ├── skills/ ← авто-детектируемые навыки └── commands/opsx/ ← slash-команды

Назначение папок

ПутьРольКто меняет
openspec/specs/Source of truth — текущие требования всей системы. По доменам.AI через archive
openspec/changes/Активная работа. Каждая задача — своя папка с 4 артефактами.AI создаёт, человек правит
openspec/changes/archive/История завершённых изменений с датой в имени.AI при /opsx:archive
openspec/config.yamlМозг проекта — стек, соглашения, правила. Инжектируется в каждый запрос.Человек
.claude/skills/SKILL.md которые Claude Code авто-детектирует как навыки.Генерирует CLI
.claude/commands/opsx/Slash-команды для AI-инструмента.Генерирует CLI

Структура одного change

ФайлНазначениеСоздаётся после
.openspec.yamlМетаданные: schema, статус артефактов, датаСразу при new/propose
proposal.mdIntent (зачем), Scope (что входит/нет), ApproachПервый
specs/*.mdДельта требований — только изменения (ADDED/MODIFIED)После proposal
design.mdАрхитектурные решения, trade-offs, диаграммыПосле proposal

Артефакты — подробно

proposal.md — Зачем и что делаем

Создаётся первым. Определяет намерение и границы задачи.

# Proposal: Add Dark Mode ## Intent Пользователи просят тёмную тему для снижения нагрузки на глаза. ## Scope ### In scope - Переключатель темы в настройках - Определение системных предпочтений (prefers-color-scheme) - Сохранение выбора в localStorage ### Out of scope - Тёмная тема для email-уведомлений ## Approach CSS custom properties + React Context для управления состоянием. ## Open questions - Нужна ли анимация при переключении?

specs/*.md — Дельта требований

Хранит только изменения. Маркеры: ADDED / MODIFIED / REMOVED. При archive мержится в основные specs.

Формат The system SHALL... — стандарт IEEE 830. SHALL = обязательно, SHOULD = рекомендуется, MAY = опционально.
# Delta for UI ## ADDED Requirements ### Requirement: Theme Selection The system SHALL allow users to choose between light and dark themes. #### Scenario: User switches to dark mode - GIVEN пользователь открыл настройки - WHEN выбирает "Dark mode" - THEN интерфейс переключается на тёмную тему - AND выбор сохраняется между сессиями

design.md — Технические решения

# Design: Add Dark Mode ## Approach CSS custom properties для тем, React Context для состояния. ## Key Decisions - CSS variables, не CSS-in-JS — нет runtime overhead - localStorage, не backend — не требует auth ## Sequence User clicks toggle → ThemeContext.toggle() → localStorage.set('theme', value) → CSS class on <html>

tasks.md — Чеклист реализации

AI читает этот файл при /opsx:apply и выполняет задачи по порядку.

# Tasks: Add Dark Mode ## Phase 1: Core Infrastructure - [ ] 1.1 Создать ThemeContext с toggle() - [ ] 1.2 Добавить CSS custom properties в globals.css - [ ] 1.3 Обернуть App в ThemeProvider ## Phase 2: UI - [ ] 2.1 Создать компонент ThemeToggle - [ ] 2.2 Добавить в Header - [ ] 2.3 Подключить localStorage ## Phase 3: Tests - [ ] 3.1 Unit-тест для ThemeContext - [ ] 3.2 E2E: тема сохраняется после reload

КомандаРольЧто делаетКогда

Expanded-профиль (дополнительно)

КомандаРольЧто делает
/opsx:new <name>ЗаготовкаТолько создаёт папку и .openspec.yaml. Артефакты не генерирует.

/opsx:verifyВалидацияПроверяет: все задачи выполнены, код соответствует specs, design отражён в коде.

/opsx:bulk-archiveПакетАрхивирует несколько завершённых изменений сразу. Разрешает конфликты автоматически.

Сценарий: пришло ТЗ → спецификация → код

1
Инициализация (один раз на проект)
openspec init → выбираешь AI-инструмент → создаётся вся структура
2
/opsx:explore — вставляешь ТЗ
AI читает кодовую базу, задаёт уточняющие вопросы, выясняет edge cases. Кода ещё нет.
3
/opsx:propose name — создаём все артефакты
AI использует контекст из explore-сессии. Создаёт proposal, specs, design, tasks за один шаг.
4
Ревью артефактов руками
Открываешь proposal.md, specs, tasks — читаешь, правишь. Просишь AI уточнить конкретный файл.
5
/opsx:apply — реализация
AI идёт по tasks.md, выполняет задачи, ставит [x]. Перед этим — очисти контекст чата.
6
/opsx:verify → /opsx:archive
Verify проверяет соответствие кода specs. Archive мержит дельту и кладёт изменение в архив.
Делай explore и propose в одной сессии — AI использует живой контекст разговора. Если начинаешь новую сессию — кратко пересказывай что решили.

openspec/config.yaml

Главный файл конфигурации. Инжектируется в каждый запрос к AI при генерации артефактов.

Три слоя

КлючЧто делаетКуда инжектируется
schemaДефолтная схема workflow. Убирает --schema из каждой команды.При каждой команде
contextСтек, архитектура, соглашения проекта. Макс. 50KB.Во ВСЕ артефакты
rulesПравила для конкретных артефактов.Только в нужный

Полный пример (Laravel)

schema: spec-driven context: | Language: Russian. All artifacts must be written in Russian. Tech stack: PHP 8.3, Laravel 11, Vue 3 + Inertia.js Database: PostgreSQL, Eloquent ORM API style: REST JSON API + web routes через Inertia Auth: Laravel Sanctum, роли через Spatie Permission Testing: Pest unit/feature, Playwright e2e Style: Laravel Pint, strict PHP types Architecture: Service Layer поверх MVC Conventions: - Backwards compatibility обязательна для всех public API - Все новые эндпоинты требуют FormRequest - Feature-тест обязателен для каждого эндпоинта rules: proposal: - Указывать нужны ли миграции и сидеры - Указывать план отката для рискованных изменений specs: - Формат Given/When/Then для всех сценариев - Покрывать авторизацию (Policy/Gate) для каждого действия design: - Указывать какие модели/relations затрагиваются - Описывать структуру миграций при изменениях схемы tasks: - Порядок: миграция → модель → сервис → контроллер → тест - Максимум 2 часа на одну задачу
Будь лаконичен — context инжектируется в каждый запрос. Только то, что реально влияет на архитектурные решения.

Создание через AI

Проанализируй кодовую базу и создай openspec/config.yaml. Прочитай: composer.json, package.json, .env.example, app/, routes/, database/migrations/ (2-3 последних). Если что-то не нашёл — пиши N/A. Артефакты пиши на русском языке.

FAQ

Это же просто waterfall?▶
Нет. Waterfall ломается из-за жёстких фаз и месяцев планирования. OpenSpec — это 10 минут на alignment перед началом. Когда что-то меняется — обновляешь spec и продолжаешь. Никаких фазовых ворот.
Чем отличается от встроенного plan mode в AI?▶
Plan mode — для одной сессии. OpenSpec — для планов которые живут несколько сессий, шерятся с командой и проходят через весь lifecycle разработки. Контекст не умирает когда закрываешь вкладку.
Можно ли на существующем проекте?▶
Да, именно для этого и создан. Specs создаются по мере появления новых задач. Не нужно описывать весь существующий код сразу — это бессмысленная трата времени.
Что если поменял AI-инструмент?▶
Specs и changes в openspec/ работают с любым инструментом — они просто Markdown-файлы. Запусти openspec update для нового инструмента.
Нужен API-ключ или MCP-сервер?▶
Нет. OpenSpec работает через slash-команды и файлы в репозитории. Просто npm-пакет и Markdown-файлы.
Как командная работа?▶
Specs и changes живут в репозитории — коллаборация через git: PR, reviews, стандартный workflow. Reviewer видит дельту требований ещё до просмотра кода.
Какую модель использовать?▶
Официальная рекомендация: Claude Opus 4.5 или GPT 5. Качество артефактов напрямую зависит от мощи модели.
OpenSpec собирает данные?▶
Только имена команд и версию пакета. Никаких путей, содержимого файлов, PII. Отключить: export OPENSPEC_TELEMETRY=0
